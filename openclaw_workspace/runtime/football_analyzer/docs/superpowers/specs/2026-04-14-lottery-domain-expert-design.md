# 足球分析系统：全玩法策略路由与知识库 (Lottery Domain Expert) 设计方案

## 1. 愿景与目标 (Vision & Goal)
在完成了底层 Agentic 架构（视觉交互、长期记忆、多空博弈、并发调度）之后，本方案将直接针对**“专业博彩投注策略”**进行革命性的升维。
我们要彻底打破系统的“胜平负偏见”，赋予它 **100% 掌握中国体育彩票三大体系（竞彩、北单、传统足彩）全部玩法** 的专家级能力。系统将从一个“预测胜负的机器”，进化为“因赛施策、寻找全盘最大 EV 漏洞”的**高级策略路由引擎 (Strategy Router)**。

---

## 2. 核心痛点与解决方案

### 2.1 痛点剖析
1. **胜平负偏见**：目前系统默认计算主胜、平局、客胜概率。但很多低赔“蚊子肉”比赛（如主胜1.15），一旦串关极易死亡，毫无投资价值。
2. **忽视高价值衍生玩法**：如果两队防守差，**总进球数**玩法可能存在极高 EV；如果主队慢热，**半全场（平胜/负胜）**玩法可能是真正的利润池。
3. **规则盲区**：AI 不懂“北单让球规则”、“竞彩单关限制”、“混合过关计算方式”，这会导致其推荐出现实中根本无法购买的实单。

### 2.2 混合驱动架构 (Math Engine + AI Expert)
我们采用 **“底层数学穷举 + 上层规则专家裁决”** 的混合模式：
1. **Math Layer (数学层)**：不再仅仅输出胜负概率，而是利用泊松分布和蒙特卡洛矩阵，**穷举计算出比分概率矩阵 (Score Matrix)**。基于此矩阵，瞬间推导出“总进球、半全场、让球胜平负”等所有衍生玩法的真实概率。
2. **Knowledge Layer (知识层)**：建立体彩三大玩法的硬性规则知识库（RAG/Prompt 注入），例如“这场比赛竞彩不开单关，只能串关”、“北单没有传统足彩的任九玩法”。
3. **Expert Layer (专家层)**：`JudgeAgent` 升级为 `Strategy Expert`，拿着数学层穷举出的全玩法 EV 表，结合知识层的规则，挑选出本场比赛**最安全且期望最高的独特玩法**。

## 3. 核心设计模块

### 3.1 模块一：全景概率引擎 (Panoptic Probability Engine)
*   **改造对象**：`QuantAgent` (基础宽客)。
*   **机制升级**：
    *   现在的泊松公式只算：`P(Home), P(Draw), P(Away)`。
    *   **升级后**：宽客必须计算出一个完整的 **N x M 比分矩阵**（例如 0:0, 1:0, 0:1 ... 3:3）。
    *   有了比分矩阵，就能精确推导：
        *   **让球胜平负**：将矩阵横移（如主让1球，则主队实际进球 - 1 后再统计胜负）。
        *   **总进球数**：统计矩阵中 i+j = 0, 1, 2, 3... 的概率和。
        *   **上下单双 (北单)**：统计总进球 >=3（上盘）和奇偶（单双）的联合概率。
*   **输出结果**：一份包含 5-10 种具体玩法的 **EV 收益预期表**。

### 3.2 模块二：体彩领域知识库 (Lottery Rulebook RAG)
*   **改造对象**：系统内存/上下文管理。
*   **机制**：创建一个硬编码或 Markdown 格式的 `lottery_rulebook.md`，包含极其严谨的体彩规则：
    *   **竞彩足球**：支持单关/串关，混合过关限制，让球规则。
    *   **北京单场**：每场必有让球（有时让整数，有时让0.5等），玩法包含胜平负、上下单双等，SP 值是浮动的（临场前未知绝对赔率，只能算预估 EV）。
    *   **传统足彩**：14场胜负、任九、6场半全场等（基于整期赛事，而非单场）。
*   **应用**：每次分析前，系统通过 `lottery_type` 参数动态将对应规则注入到 `JudgeAgent` 的上下文。

### 3.3 模块三：策略路由器 (Strategy Router)
*   **改造对象**：`JudgeAgent` 的 Prompt 与决策树。
*   **机制**：法官的决策逻辑从“买不买胜平负”，变成**“买什么玩法最划算”**。
*   **决策树示例**：
    *   如果这场是焦点战（如皇马 vs 巴萨），胜平负赔率极度内卷，EV = -0.05。
    *   法官查看宽客的“比分矩阵”，发现两队防守近期极差（Scout 情报证实主力中卫缺阵），推导出“总进球 >= 3”的概率高达 70%，而机构开出的“总进球 3, 4”赔率极高，EV = +0.15。
    *   法官最终裁决：**“放弃胜平负玩法。推荐玩法：【总进球数 3球/4球】。理由：防守伤停导致进球期望飙升，此玩法存在机构定价漏洞。”**

## 4. 实施路径 (Implementation Plan)

### Phase 5.1: 数学引擎扩容 (Score Matrix & Derived Probabilities)
*   编写一个纯 Python 工具类 `LotteryMathEngine`，接收两队的预期进球数（xG），返回一张详尽的《衍生玩法概率表》（含让球、进球数、半全场等）。

### Phase 5.2: 知识库植入 (Domain Knowledge Injection)
*   编写 `docs/lottery_rulebook.md`，将三大玩法的死规则固化。
*   在 `SyndicateOS` 启动分析时，将相关规则直接塞给法官。

### Phase 5.3: 策略路由器实战 (Strategy Router Integration)
*   重写 `JudgeAgent` 的 System Prompt。强迫它：**“绝对不要只盯着胜平负。如果低赔胜平负没有价值，你必须在其他玩法中寻找 Alpha，或者建议合理的串关策略。”**
